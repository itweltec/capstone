Project Scope

The Mountain is a proof of concept of what we can achieve and is in no way a finished product. We wanted the user to be able to see the core functionality of the game such as movement, time lapses and the energy system while also being able to appreciate the amazing scenery. We knew that we wouldn't have time to create everything we wanted to an acceptable standard, and so we choose only the things that we deemed critical game features or core aspects of the environment. The rest of the game's features were assigned to the second development iteration, which will take place after this course.

DEVELOPMENT

1. PLANNING

To begin the project we needed a concrete idea of what we wanted to create, multiple planning processes, and a solid set of tools and methodologies for our team to work efficiently. This groundwork became a valuable asset for the success of the project, and allowed us to develop the working concept of the game in a suitable time frame.

2. FIRST STAGES OF DEVELOPMENT

The initial game outline required us to quickly develop the fundamentals we'd need to create our game including camera control, character movement and user interaction. We developed these core features iteratively and continually tested the game throughout development to gather personal testing data for future tweaking.

3. REFINING OF MODULES

The main drive for the project was our iterative development approach. This approach allowed our team to work efficiently, create basic models and iteratively upgrade them as we saw fit. Throughout the project we integration tested our game and performed qualitative summaries of each module to determine if they needed to be rebuilt or worked on further. This allowed us to further refine processes and modules in the later development stages, thus ensuing quality end-results for each module.

4. INTEGRATION

Once we had our individual components completed, the game needed to be compiled together into one cohesive unit. Integrating the game like this surfaced many gaps in the project tasks that we needed to fill, this was because of our use of the top-down planning model which limited our ability to foresee missing detail in the project plan. We sewed the seems shut and compiled our work together into the final version of The Mountain tech demo.
